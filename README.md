# CS-255
The project. The client was DriverPass, a startup helping students pass the DMV exam. I designed a cloud-based system with online classes, practice tests, and driving lesson scheduling, supporting five user roles plus DMV and payment integrations.
What I did well. I tied every design choice to something the client actually said in the interview — cloud hosting, offline read-only access, the activity log, role-based permissions. Nothing was invented; everything traces back to a specific need.
What I'd revise. The "Schedule a Driving Lesson" activity diagram. I combined slot availability and driver/car assignment into one decision, but those are really two separate failure points. I'd split them and add swim lanes for the customer vs. the system.
Interpreting user needs. I read the interview twice — first for context, then to pull out concrete requirements as direct quotes. Anything the client didn't address, I flagged as an assumption instead of guessing. User needs matter because a clever design that misses the actual problem is a failed design — and that failure usually doesn't surface until it's expensive to fix.
My design approach. Outside-in: actors first, then use cases, then data and behavior, then technology. Each UML diagram forces a different lens, and contradictions between them expose gaps. Going forward, I'd add measurable nonfunctional targets and check in with the client more often instead of presenting a finished design at the end.
